# Coral Context: Detailed Engineering Specification

**Status:** Current (Zig/SQLite implementation, Milestone 3-5 in progress)
**Last Updated:** March 2026
**Architecture:** SQLite backend with Zig deterministic core

---

## Executive Summary

Coral Context is a neurosymbolic orchestration framework optimized for edge environments. It systematically decouples probabilistic reasoning (LLMs) from deterministic execution (DAGs and Graph Databases). By utilizing Zig's high-performance, low-memory footprint alongside embedded SQLite, the system delegates workflow routing, knowledge retrieval, and tool orchestration to a strictly typed engine. LLMs are treated solely as unstructured data compilers accessed over HTTP, preventing hallucinations in the critical path and enabling complex agentic behaviors on devices as small as a Raspberry Pi.

**Core Technologies:**
- **Zig** for bitwise, deterministic DAG execution
- **SQLite** (embedded relational database) for semantic graphs and recursive CTE traversal
- **Extism** for secure WASM sandboxing
- **HTTP LLMs** for probabilistic inference

## 1. Concepts and Design Decisions

### 1.1 Directed Acyclic Graphs (DAG) & Targets as Units of Work

A **Directed Acyclic Graph (DAG)** is a mathematical structure representing a network of nodes connected by directed edges, with no circular dependencies (cycles). In Coral Context, the DAG is the deterministic routing engine that replaces the unpredictable reasoning steps of a probabilistic LLM.

**DAG Targets as Units of Work:** Derived from the Python `toolcore`/`yamake` legacy, a "Target" is the fundamental primitive of execution. A Target is a unified abstraction that simultaneously serves as a CLI subcommand, an interactive REPL command, a parallel worker task, and a node in a workflow graph. By defining a Target through what it *depends on* and what it *provides*, the system uses topological sorting (Kahn's algorithm) to automatically determine the exact sequence of tools required to accomplish a user's goal.

### 1.2 TraitSets & Compile-Time vs. Runtime Target Representations

A **TraitSet** is the mechanism the system uses to implement "duck typing" for Target capabilities. Instead of hard-coding that "Target A requires Target B," a Target specifies that it requires an abstract capability (e.g., "requires C++ compiler").

Because the engine is written in Zig, these string-based abstract traits are internally converted into hyper-fast bitvectors:

- **Compile-Time Representation (`std.EnumArray`, `std.EnumSet`, `IntegerBitSet`):** For built-in system tools where the total ontology of capabilities is known at compile time, traits are mapped to a Zig `enum`. `std.EnumSet` creates a strictly typed bitvector where each tag is a bit. `std.EnumArray` acts as the execution registry, guaranteeing perfectly contiguous memory lookups mapping the enum to the Target's function pointer. The distance between required and provided traits is calculated in a single CPU cycle using `@popCount` (Hamming weight).

- **Runtime Representation (`DynamicBitSet`):** When targets are loaded dynamically (e.g., from JSON or generated by an LLM), the total number of unique traits isn't known at compile time. The system uses a `StringInterner` to assign an integer ID to new strings, and allocates a `std.bit_set.DynamicBitSet` on the heap exactly wide enough to hold the masks. This allows the runtime engine to execute the exact same hardware-accelerated bitwise masking logic as the compile-time engine.

### 1.3 ContextNodes & Level of Detail (LOD)

A **ContextNode** is the universal semantic data structure in the system. It represents ingested entities (from knowledge bases like YAGO 4.5), serialized DAG Targets, or cached blocks of LLM reasoning.

To maximize the density of an LLM's context window without exceeding token limits, ContextNodes utilize **Level of Detail (LOD)**. Borrowed from 3D game engines—where objects lose polygons the further they are from the camera—ContextNodes shed text length based on their graph distance from the semantic center of the user's prompt:

| LOD | Size | Purpose |
|-----|------|---------|
| LOD 0 | Full text | The complete semantic text. Used only for immediate focal points of a query. |
| LOD 1 | ~800 chars | A comprehensive summary. |
| LOD 2 | ~240 chars | A brief abstract. |
| LOD 3 | ~80 chars | A tiny snippet or definition. |
| LOD 4 | Name only | Only the brief name or UUID of the entity. |
| LOD 5 | Keywords/Hashtags | Fast-resolution aliases to bypass heavy vector search embeddings for direct textual matches. |

### 1.4 QueueReactor & Stack Frame Execution Model

The **QueueReactor** is the Actor-model backbone that provides the system's "fearless concurrency" without a garbage collector or Rust-style borrow checker.

Each QueueReactor manages an isolated thread pool and a multi-producer/single-consumer (MPSC) event queue. It handles concurrency by modeling task execution on the concept of a **C++ Stack Frame**. In C++, local variables are pushed to the stack and automatically destroyed when a function returns. In Zig, whenever a worker thread pulls a task from the queue, it is assigned a dedicated `std.heap.ArenaAllocator`. All dynamic memory required to execute that Target is allocated inside this Arena. When the task finishes, the entire Arena is dropped at once. Threads never share mutable memory, meaning the "heap" is deterministically popped just like a stack frame, eliminating memory fragmentation and data races.

### 1.5 WASM Tools & Evolutionary Sandboxes (Extism)

Because Coral Context treats LLMs as untrusted compilers, any dynamic logic generated by an LLM must be isolated. This is achieved using WebAssembly (WASM) and the **Extism** runtime sandbox.

- **Human/AI-Supervised Test-Driven Development (TDD):** If the DAG engine encounters a gap in its capabilities, it asks an LLM to write a new tool in a compilable language. This tool is compiled to WASM and placed in the Extism sandbox, which strictly prevents unauthorized access to the host's filesystem, network, or memory.

- **Evolutionary Algorithms:** The system passes test data into the WASM sandbox. If the LLM-generated tool successfully bridges the data gap (e.g., properly transforming unstructured text into structured ContextNodes), the tool "survives." Its WASM binary is cached permanently in SQLite, and it becomes a verified, deterministic node in the DAG for future use.

## 2. Storage Architecture (Milestone 2)

### 2.1 SQLite as Unified Backend

**Rationale**: SQLite provides a single embedded database supporting:
- Recursive CTE queries for graph traversal (replaces Cypher/Datalog)
- Embedding BLOBs as raw `f32` arrays; KNN computed in Zig via cosine similarity
- Temporal validity ranges via `valid_from`/`valid_to` columns
- Persistent storage backend selection: `mem` (`:memory:`, ephemeral), `file` (edge/server)

**Build Configuration**:
```bash
zig build                          # links vendored SQLite (single .c file, SHA256-verified)
zig build -Doptimize=ReleaseSmall  # ~1 MB binary including SQLite
```

SQLite is vendored as a single amalgamation `.c` file. No external shared library or linker path is required beyond the standard C runtime.

### 2.2 Schema Definition (SQL DDL)

All tables are created via `src/coral/schema.zig` using standard SQL DDL. Each statement is idempotent (`IF NOT EXISTS`).

#### ContextNode Table (§2.2)
```sql
CREATE TABLE IF NOT EXISTS context_nodes (
    id INTEGER PRIMARY KEY,
    lod0 TEXT NOT NULL DEFAULT '',
    lod1 TEXT NOT NULL DEFAULT '',
    lod2 TEXT NOT NULL DEFAULT '',
    lod3 TEXT NOT NULL DEFAULT '',
    lod4 TEXT NOT NULL DEFAULT '',
    lod5 TEXT NOT NULL DEFAULT '',
    embedding BLOB NOT NULL DEFAULT x'',
    valid_from REAL NOT NULL DEFAULT 0.0,
    valid_to REAL,
    confidence INTEGER NOT NULL DEFAULT 0,
    provenance_id INTEGER NOT NULL DEFAULT 0
)
```

**Key Design**:
- **Primary key**: `id INTEGER PRIMARY KEY` — SQLite ROWID alias, maps to `i64` in Zig
- **LOD pyramid**: Six levels (`lod0`–`lod5`) for context window packing at different graph distances
- **Embeddings**: Raw BLOB of little-endian IEEE 754 `f32` values (4 bytes each); KNN computed in Zig via cosine similarity, **not** via external ANN index (correct for ≤100K nodes at edge scale)
- **Temporal**: `valid_from`/`valid_to` for explicit validity ranges (Unix timestamps as REAL)

#### Target Table (§2.3)
```sql
CREATE TABLE IF NOT EXISTS targets (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    depends_words BLOB NOT NULL DEFAULT x'',
    provides_words BLOB NOT NULL DEFAULT x'',

    is_essential INTEGER NOT NULL DEFAULT 0
)

CREATE INDEX IF NOT EXISTS targets_by_name ON targets(name)
```

**Key Design**:
- **Trait bitmasks**: `depends_words`/`provides_words` encode the raw `usize` word array of a `DynamicBitSetUnmanaged` (native byte order)
- **Index**: `targets_by_name` enables O(1) lookup by name in DAG resolution

### 2.3 SQL Recursive CTE Query Templates

**Transitive Dependency Traversal** (replaces Datalog recursive rules):
```sql
WITH RECURSIVE tr(id) AS (
    SELECT to_id FROM depends_on WHERE from_id = ?1
    UNION
    SELECT d.to_id FROM depends_on d JOIN tr ON tr.id = d.from_id
)
SELECT t.id, t.name, t.depends_words, t.provides_words, t.total_bits
FROM tr JOIN targets t ON t.id = tr.id
```

**BFS Neighbor Traversal** (for LOD routing):
```sql
WITH RECURSIVE bfs(id, dist) AS (
    SELECT ?1, 0
    UNION
    SELECT n.to_id, b.dist + 1
    FROM bfs b JOIN neighbor_of n ON n.from_id = b.id
    WHERE b.dist < 10
)
SELECT b.id, b.dist, cn.lod4
FROM bfs b JOIN context_nodes cn ON cn.id = b.id
ORDER BY b.dist
```

Both queries are defined as string constants in `src/coral/schema.zig` (`QUERY_TRANSITIVE_DEPS` and `QUERY_NEIGHBOR_BFS`).

## 3. Data Types & Memory Management (Zig Implementation)

### 3.1 NodeId: SQLite INTEGER Row ID

SQLite's `INTEGER PRIMARY KEY` is a 64-bit signed integer (ROWID alias). In Zig the type is `i64` as returned by `sqlite3_last_insert_rowid()`. New nodes receive auto-assigned row IDs; there is no UUID split across two columns.

```zig
// Insert a node, get its auto-assigned id back
try stmt.exec(.{});
const node_id: i64 = db.lastInsertRowId();
```

**Invariant**: IDs are monotonically increasing integers. For stable cross-device identity, use the `lod5` (keyword) column to store an external IRI or UUID string.

### 3.2 ContextNode: The Semantic Entity

```zig
pub const ContextNode = struct {
    id: i64,
    lod: [6][]const u8,   // lod[0]=full … lod[5]=keywords
    lod_owned: u8,        // bitmask: bit i set → caller owns lod[i]
    embedding: []const f32,
    valid_from: f64,      // Unix timestamp
    valid_to: ?f64,       // null = no expiry
    confidence: i64,
    provenance_id: i64,
};
```

**Lifetime Contract**: Strings are duped from SQLite column text before the statement step is reset. Callers track ownership via the `lod_owned` bitmask; bits set to 1 require `allocator.free(lod[i])` on teardown.

### 3.3 KnnHit: Search Result

```zig
pub const KnnHit = struct {
    id: i64,
    lod4: []const u8,   // name-level text for ranking display
    distance: f32,      // 0.0 = identical, 2.0 = opposite
};
```

### 3.4 EdgeType: Topology Classification

```zig
pub const EdgeType = enum(i16) {

    semantic_similarity = 3,
```

## 4. Runtime Operations (Milestone 2-3)

### 4.1 Library: The Database Handle

```zig
pub const Library = struct {
    allocator: std.mem.Allocator,
    db: *sqlite3,
    mutex: std.Thread.RwLock,   // write-exclusive, read-shared (WAL mode)

    pub fn init(allocator, engine: StorageEngine, path: []const u8) !*Library
    pub fn deinit(self: *Self) void
    pub fn initSchema(self: *Self) !void

    // Node CRUD
    pub fn insertNode(self: *Self, node: ContextNode) !void
    pub fn fetchNode(self: *Self, id: i64) !?ContextNode
    pub fn freeNode(self: *Self, node: ContextNode) void

    // Target CRUD
    pub fn insertTarget(self: *Self, name: []const u8,
                        depends_words: []const usize,
                        provides_words: []const usize,
                        total_bits: usize,
                        is_essential: bool) !i64

    // Edge CRUD
    pub fn insertDependsOn(self: *Self, from_id: i64, to_id: i64) !void
    pub fn insertNeighborOf(self: *Self, from_id: i64, to_id: i64,
                            distance: f32, edge_type: EdgeType) !void

    // Graph traversal (recursive CTE)
    pub fn traverseFrom(self: *Self, root_id: i64) ![]TraversalRow
    pub fn bfsNeighbors(self: *Self, center_id: i64) ![]BfsRow

    // Pipeline factories
    pub fn createHydrationPipeline(self: *Self) HydrationPipeline
    pub fn createContextPacker(self: *Self, max_tokens: usize) ContextPacker
};
```

**Thread safety**: `mutex` is taken write-exclusive for all `INSERT`/`UPDATE` statements and read-shared for `SELECT` statements (SQLite WAL mode allows concurrent readers).

### 4.2 HydrationPipeline: KNN + Edge Persistence

**3-step pipeline**:
1. **Accept** embedding vector from edge LLM
2. **KNN**: Fetch all embeddings from SQLite (`embedding BLOB`), deserialize `f32` arrays, compute cosine distance in Zig, select top-K
3. **Persist**: Insert `NEIGHBOR_OF` edges with distance weights into SQLite

```zig
pub const HydrationPipeline = struct {
    allocator: std.mem.Allocator,
    library: *Library,

    /// Cosine distance (0.0 = identical, 2.0 = opposite)
    fn cosineDistance(a: []const f32, b: []const f32) f32

    /// Step 2: Fetch all embeddings, compute distances, return top-K sorted
    pub fn knnSearch(self: *Self, query_vec: []const f32, k: usize) ![]KnnHit

    /// Step 3: Persist KNN edges into SQLite
    pub fn persistEdges(self: *Self, source_id: i64, knn_hits: []const KnnHit) !void

    /// Full pipeline: KNN → persist
    pub fn hydrate(self: *Self, source_id: i64, query_vec: []const f32, k: usize) !usize
};
```

**Key Design**:
- **No external ANN index**: Correct and efficient for ≤100K nodes at edge scale.
- **In-Zig cosine**: SIMD-friendly dot product + norm product computed in tight loop.
- **Deterministic**: No randomness; exact reproducibility across runs.

### 4.3 ContextPacker: LOD Selection & Context Window

```zig
pub const ContextPacker = struct {
    allocator: std.mem.Allocator,
    library: *Library,
    max_tokens: usize,
    chars_per_token: usize = 4,

    /// Select LOD index by graph distance from semantic center
    pub fn selectLodIndex(graph_distance: u32) usize

    /// SQL recursive CTE BFS from semantic center
    pub fn getNodesByDistance(self: *Self, semantic_center_id: i64) ![]BfsRow

    /// Pack context respecting token budget
    pub fn pack(self: *Self, semantic_center_id: i64) ![]const u8
};
```

**LOD Mapping**:
- Distance 0 (semantic center): `lod[0]` (complete text)
- Distance 1: `lod[1]` (800 chars) or fallback to `lod[0]`
- Distance 2: `lod[2]` (240 chars) or fallback to `lod[4]`
- Distance ≥3: `lod[4]` only

**Token Budget**: Estimate via `text.len / 4` (conservative); downgrade to `lod[4]` if token budget exceeded.

## 5. Binary IPC: ContextNode Serialization

### 5.1 Zero-Copy Schema (FlatBuffers-style)

**BinaryHeader** (precedes all payloads):
```zig
pub const BinaryHeader = extern struct {
    magic: [4]u8 align(1),           // "CCND"
    version: u32 align(1),           // 1
    payload_type: PayloadType align(1),
    payload_size: u32 align(1),
    checksum: u32 align(1),
};
```

**BinaryContextNode** (for WASM IPC):
```zig
pub const BinaryContextNode = extern struct {
    header: BinaryHeader align(1),
    id: i64 align(1),
    valid_from_ts: i64 align(1),
    valid_to_ts: i64 align(1),
    // Offsets into variable-length string region
    lod0_offset: u32 align(1),
    lod0_len: u32 align(1),
    lod1_offset: u32 align(1),
    lod1_len: u32 align(1),
    // ... (lod2–lod5 similarly)

    pub fn init(node: *const ContextNode, buffer: []u8) BinaryContextNode
    pub fn getLod(self: *const BinaryContextNode, idx: usize, buffer: []const u8) []const u8

```

**Key Design**:
- **Zero-copy**: Strings referenced by offset/length into a single contiguous buffer.
- **No JSON**: Binary only; JSON parsing forbidden across WASM boundary to maintain microsecond latency.
- **Fixed-size header**: Enables deterministic buffer allocation.

## 9. Deployment Profiles

### 9.1 Edge (Raspberry Pi, ARM32/64)

**Configuration**:
- Storage: SQLite file (persistent, low I/O, WAL mode)
- Max nodes: ≤100K (KNN in-Zig is correct at this scale)
- LLM: Local edge LLM (ollama/llama.cpp on same device or nearby)

**Footprint**: ~1 MB binary (Zig ReleaseSmall) + SQLite database file

### 9.2 Server (Linux x86_64)

**Configuration**:
- Storage: SQLite file (WAL mode, concurrent reads)
- Max nodes: ≤1M (at this scale consider an external ANN index for KNN; in-Zig linear scan becomes a bottleneck)
- LLM: Remote cloud API or high-performance frontier model

**Scalability**: Add external HNSW/Annoy index service; Coral Context queries it for top-K, then loads full nodes from SQLite.

## 10. Known Limitations & Workarounds

### 10.1 No Native ANN Index

**Issue**: SQLite has no HNSW or other approximate nearest-neighbor index.
**Workaround**: Fetch all `embedding` BLOBs, deserialize `f32` arrays, compute cosine in Zig, sort top-K.
**Correct for**: ≤100K nodes at edge scale.
**Workaround for server scale**: External ANN service (Milvus, Weaviate, Pinecone) or LanceDB integration.

### 10.2 Embedding BLOB Byte Order

**Issue**: Embeddings are stored as raw little-endian `f32` BLOBs. On big-endian hosts the bytes would need swapping.
**Workaround**: All supported targets (x86_64, aarch64, riscv64) are little-endian; no swap needed.
**Impact**: Document the byte order invariant and gate cross-architecture reads behind a compile-time check if big-endian support is ever added.

### 10.3 SQLite Write Serialization

**Issue**: SQLite allows only one writer at a time, even in WAL mode.
**Workaround**: `Library.mutex` is a `RwLock`; write-exclusive for `INSERT`/`UPDATE`, read-shared for `SELECT`. Concurrent reads are fully parallel.
**Impact**: Write throughput bounded by single-writer; batch ingestion via `BatchIngestor` amortizes this with large transactions.

### 10.4 String Lifetime from SQLite Columns

**Issue**: `sqlite3_column_text()` returns a pointer valid only until the next step or statement reset.
**Workaround**: All column strings are duped via `allocator.dupe()` before the statement is stepped or reset (see `Library.fetchNode()`).
**Impact**: All callers must call `freeNode()` (or track `lod_owned` bits) after use or leak memory.

## 12. Implementation Milestones

### Milestone 1: The Deterministic Core (`yamake-zig`)

*The foundation. Porting the Python DAG engine to a compiled, strictly typed Zig binary.*

**Capabilities:**
- Universal CLI and REPL execution (utilizing `zig-clap` and the `linenoise` C-binding)
- Makefile-like task execution and dependency resolution using Kahn's algorithm
- "Duck-typing" dependency resolution via capability sets (TraitSets)
- Hardware-accelerated `@popCount` distance logic to find the closest matching tools for a given context

**Data Ingestion:**
- Static JSON files defining targets, dependencies, and provided capabilities

**Technical Specification:**
- **Target Registry:** Dual-registry system: `comptime` ArrayRegistry for baked-in system commands, runtime `StringHashMap` registry for dynamically parsed JSON tools
- **TraitSets:** `std.bit_set.ArrayBitSet` for compile-time logic, `std.bit_set.DynamicBitSet` + `StringInterner` for resolving JSON dependencies into integer masks

### Milestone 2: Semantic Graph Integration

*The relational overlay. Moving beyond simple bitmasks to complex, multi-hop property graphs.*

**Capabilities:**
- Embedded SQLite database directly within the Zig application
- Recursive CTE queries for graph traversal (replaces Cypher/Datalog `MATCH` patterns)
- Data-Oriented Design (DOD) structuring of ContextNodes

**Data Ingestion:**
- Pre-processed CSV or Parquet files (e.g., flattened YAGO 4.5 RDF datasets)
- JSON configurations for runtime-defined nodes

**Technical Specification:**
- **Hybrid Resolution:** Use SQL recursive CTEs to filter the graph topologically, returning row IDs back to Zig for final bitwise `@popCount` validation against DynamicBitSet trait masks

### Milestone 3: Context Packing & Semantic Memory

*The semantic memory. KNN search and temporal validity.*

**Capabilities:**
- K-Nearest Neighbor (KNN) semantic search to find the "center" of a user's prompt
- Temporal validity via `valid_from`/`valid_to` column ranges
- **LOD Context Packing:** Dynamically shrinking or expanding text passed to an LLM based on graph distance from semantic center
- **Streaming batch ingestion** via `BatchIngestor` and fluent `IngestBuilder` API

**Data Ingestion:**
- Streaming Turtle (TTL) ingestion pipelines via `src/coral/batch.zig`

**Technical Specification:**
- **The Hydration Loop:**
  1. **Embed query** and execute KNN search (in-Zig cosine similarity over `embedding` BLOBs) — returns `(id, lod4, distance)` tuples
  2. **Construct `NEIGHBOR_OF` edges** in SQLite using the distance as edge weights
  3. **SQL recursive CTE BFS** computes graph distances for LOD routing
- **LOD Algorithm:** Threshold logic allocating `lod[0]` to immediate neighbors and degrading to `lod[4]` for distant nodes within the token budget

### Milestone 4: WebAssembly Sandboxing (Extism)

*The execution sandbox. Executing dynamic, LLM-generated logic securely.*

**Capabilities:**

**Technical Specification:**

*The AI orchestrator. Wiring the HTTP LLM fallback and surfacing the API.*

## 13. Ingestion Constraints

Coral Context is explicitly designed as a *runtime engine*. While it relies heavily on complex data sources (like the YAGO 4.5 ontology or DPO datasets), the Zig engine does **not** perform offline ETL (Extract, Transform, Load) operations for bulk sources.

Raw knowledge sources (YAGO Turtle files, CSV, Parquet) are ingested at runtime via `BatchIngestor` using the streaming RDF parser pipeline:

```
TTL file → Lexer → Parser → TripleMapper → BatchIngestor → Library (SQLite)
```

For very large datasets (>1M triples) the recommended pattern is to pre-process into smaller per-entity Turtle files using the Python `guidance-py` provider, then ingest incrementally. The `max_triples` and `skip_errors` options in `BatchConfig` allow partial ingestion for testing.

## Appendix A: SQLite C API Quick Reference

```c
// Open database (":memory:" for ephemeral)
int rc = sqlite3_open(path, &db);

// Prepare statement
sqlite3_stmt *stmt;
rc = sqlite3_prepare_v2(db, sql, -1, &stmt, NULL);

// Bind parameters (1-indexed)
sqlite3_bind_int64(stmt, 1, row_id);
sqlite3_bind_text(stmt, 2, text, -1, SQLITE_STATIC);
sqlite3_bind_blob(stmt, 3, data, len, SQLITE_STATIC);

// Step / fetch rows
while ((rc = sqlite3_step(stmt)) == SQLITE_ROW) {
    sqlite3_int64 id  = sqlite3_column_int64(stmt, 0);
    const char *text  = (const char*)sqlite3_column_text(stmt, 1);
    const void *blob  = sqlite3_column_blob(stmt, 2);
    int blob_len      = sqlite3_column_bytes(stmt, 2);
}

// Reset for reuse
sqlite3_reset(stmt);

// Finalize (must call for every prepare)
sqlite3_finalize(stmt);

// Close
sqlite3_close(db);
```

**Notes**:
- `sqlite3_column_text()` / `sqlite3_column_blob()` pointers are valid only until the next `sqlite3_step()` or `sqlite3_reset()` — always `dupe` before stepping.
- Use `SQLITE_STATIC` (null function pointer) instead of `SQLITE_TRANSIENT` when the buffer outlives the bind call; avoids a redundant copy.
- WAL mode: `PRAGMA journal_mode=WAL` enables concurrent readers with a single writer.

## Appendix B: SQL Recursive CTE Quick Reference

**Basic recursive traversal**:
```sql
WITH RECURSIVE path(id) AS (
    SELECT ?1              -- seed: starting node
    UNION ALL
    SELECT e.to_id
    FROM path p JOIN edge e ON e.from_id = p.id
)
SELECT id FROM path
```

**Depth-limited BFS with distance**:
```sql
WITH RECURSIVE bfs(id, dist) AS (
    SELECT ?1, 0
    UNION
    SELECT e.to_id, b.dist + 1
    FROM bfs b JOIN neighbor_of e ON e.from_id = b.id
    WHERE b.dist < 10      -- cap depth to prevent runaway queries
)
SELECT b.id, b.dist, cn.lod4
FROM bfs b JOIN context_nodes cn ON cn.id = b.id
ORDER BY b.dist
```

**Key rules**:
- `UNION` (not `UNION ALL`) eliminates cycles by deduplicating visited nodes.
- The recursive term must reference the CTE name exactly once.
- SQLite enforces the `WHERE` guard before recursing; always add a depth cap.
- Aggregates (`COUNT`, `MIN`) cannot appear in the recursive term.
