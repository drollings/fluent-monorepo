# Coral Context: Architectural Design Document

**Abstract:** Coral Context is a neurosymbolic Artificial Intelligence orchestration framework designed for extreme edge efficiency (e.g., Raspberry Pi). It strips non-deterministic logic, routing, and tool orchestration away from probabilistic LLMs, shifting those workloads into a deterministic, bitwise Directed Acyclic Graph (DAG) executed in Zig. LLMs are relegated to the role of unstructured data compilers over HTTP, while the core system resolves semantic relationships using an embedded SQLite database and executes logic securely via WebAssembly (Extism).

---

## 1. Core Architectural Components

### 1.1 Concurrency Model: The QueueReactor Pattern

Coral Context relies on an Actor Model-inspired concurrency architecture to achieve "fearless concurrency" without a borrow checker.

* **The `QueueReactor`**: Workloads are isolated into distinct queues. Each `QueueReactor` manages its own `std.Thread.Pool` and a thread-safe MPSC (Multi-Producer, Single-Consumer) channel.
* **Arena Isolation**: Tasks submitted to a `QueueReactor` are packaged with a dedicated `std.heap.ArenaAllocator`. Threads never share mutable memory; upon task completion, the worker thread instantly frees the entire Arena, eliminating data races and memory leaks.

### 1.2 The Semantic Unit: ContextNodes & ECS

Data is structured using a Data-Oriented Design (DOD) approach mirroring an Entity Component System (ECS).

* **`ContextNode` (The Entity)**: Identified by a SQLite row ID (`i64`) and LOD columns for context window packing:
  * `lod0`: Complete semantic text
  * `lod1`: 800-character max summary
  * `lod2`: 240-character max brief
  * `lod3`: 80-character max snippet
  * `lod4`: Brief identifier (name)
  * `lod5`: Keywords / hashtags

* **Helper Components**: Data stored in associated SQLite tables, loaded only when necessary:
  * `EmbeddingCache`: Float vector stored as a raw `f32` BLOB in `context_nodes.embedding`; KNN computed in Zig via cosine similarity.
  * `NodeNeighbor`: Scene graph edges (distance/type) stored in the `neighbor_of` SQLite table.
  * `TimeIndex`: Temporal validity via `valid_from` / `valid_to` REAL columns; range queries use standard SQL `WHERE` predicates.
  * `Provenance` & `Reputation`: Source tracking via `provenance_registry`; alignment scores via `confidence` column.

### 1.3 Storage: SQLite (Unified Backend)

Coral Context uses **SQLite** as the single persistent backend. SQLite is vendored as a single-file C amalgamation and compiled into the binary — no external shared library required.

* **Node payloads** (`context_nodes`): LOD text pyramid and float embedding BLOB stored in SQLite.
* **Target DAG** (`targets`): Target definitions with DynamicBitSet trait masks stored as `depends_words`/`provides_words` BLOBs.
* **Graph edges** (`depends_on`, `provides_capability`, `neighbor_of`): All topology stored in SQLite tables, traversed via SQL recursive CTEs.
* **WASM tool cache** (`wasm_tools`): Compiled `.wasm` binaries stored as base64-encoded text.
* **Temporal validity**: `valid_from`/`valid_to` REAL columns; filtered by standard SQL range predicates.
* **KNN search**: Embedding BLOBs fetched from SQLite; cosine similarity computed in Zig (correct at edge scale ≤100K nodes).

Storage backend is selectable: `mem` (`:memory:` SQLite, testing/ephemeral) or `file` (on-disk SQLite, edge/server).

### 1.4 WASM Execution & Binary IPC

Executable tools, whether hardcoded or generated by LLMs, run inside **Extism** WebAssembly sandboxes.

* **Binary IPC**: To maintain microsecond latency, Zig and Extism WASM plugins communicate exclusively via binary serialization (FlatBuffers-style offset/length encoding). JSON parsing is strictly forbidden across this boundary.
* **Security & Portability**: The sandbox prevents untested, LLM-generated tools from crashing the host or leaking memory.

### 1.5 System Boundaries & Ingestion

* **External APIs**: The system acts as a Model Context Protocol (MCP) server. External communication (HTTPS/stdio) uses JSON, which is immediately parsed into Zig binary structs at the perimeter.
* **LLM Integration**: Edge or frontier LLMs are accessed exclusively via HTTP (e.g., to an external Ollama/llama.cpp server or cloud API).
* **Ingestion Pipelines**: YAGO 4.5 RDF triples and other structured datasets are ingested at runtime via `BatchIngestor` using the Turtle streaming parser pipeline (`TTL → Lexer → Parser → TripleMapper → Library`).

## 2. Implementation Milestones

### Milestone 1: The Deterministic Core (`yamake-zig`)

*Objective: Establish the Zig execution baseline by porting the Python `toolcore`/`yamake` engine, focusing on bitwise operations and the Universal Target Pattern.*

* **Target Registry & Execution**: Replace Python's dynamic decorators with a `comptime` Array Registry for built-in system tools, and a Zig struct-based runtime registry using function pointers for dynamic tasks.
* **TraitSets & Duck Typing**: Implement `IntegerBitSet` (comptime/fixed size) and `DynamicBitSet` (runtime parsing) to handle DAG logic.
* **Distance Logic**: Implement hardware-accelerated `@popCount` masking to measure the distance between a target's required traits and the runtime's available traits.
* **Interfaces**: Implement a nested CLI command parser and a REPL using the `linenoise` C-binding.
* **Forward Compatibility**: Structure the internal Target ID system using `i64` (SQLite ROWID); trait masks encoded as `DynamicBitSetUnmanaged` word arrays in BLOBs.

### Milestone 2: The Semantic Graph (SQLite Integration)

*Objective: Embed SQLite to manage dependency graphs and semantic relationships that exceed simple bitmask boundaries.*

* Bind the SQLite C API to Zig via `src/coral/db.zig`.
* Define SQL DDL for `context_nodes`, `targets`, `depends_on`, `provides_capability`, `neighbor_of`, and `wasm_tools` (see `src/coral/schema.zig`).
* Implement SQL recursive CTEs for transitive dependency traversal, passing row IDs and BLOB-encoded TraitSet words back to Zig for final `@popCount` capability verification.

### Milestone 3: Context Packing & Semantic Search

*Objective: Connect the deterministic graph to vector embeddings and the LOD packing algorithm.*

* Implement the embedding fetch loop: store `f32` arrays as BLOBs in `context_nodes.embedding`, pull candidates, compute cosine distance in Zig, select top-K.
* Build the Hydration Pipeline:
  1. Receive semantic center embedding from edge LLM.
  2. KNN search: fetch all embedding BLOBs from SQLite, rank in Zig via cosine similarity.
  3. Persist `neighbor_of` edges in SQLite using cosine distances as weights.
  4. SQL recursive CTE BFS traversal computes graph distances for LOD routing.
* Streaming batch ingestion via `BatchIngestor` fluent API.

### Milestone 4: WASM Sandboxing via Extism

*Objective: Replace static Zig function pointers with dynamic WASM binaries.*

* Embed the Extism runtime via C-bindings.
* Implement the FlatBuffers-style binary serialization layer to pass ContextNodes into and out of WASM memory.
* Execute verified compilation of a tool into WASM and trigger it dynamically from a resolved DAG node.

### Milestone 5: The Neurosymbolic Engine & MCP Server

*Objective: Wire the probabilistic fallback and surface the system to the outside world.*

* Build the MCP JSON-RPC perimeter router.
* Implement the LLM HTTP fallback loop: If a DAG target is unsatisfied, query the external HTTP LLM to generate missing structured data or emit WASM code to fulfill the missing logic.
* Complete the lifecycle by inserting newly confirmed LLM reasoning back into SQLite as a cached tool/ContextNode (`persistSolution` in `QueueReactor`).
* L4.5 decompose tier: `LocalDecomposer` breaks complex tasks into sub-tasks, routes each recursively, deduplicates results, and persists novel solutions as new ContextNodes.
